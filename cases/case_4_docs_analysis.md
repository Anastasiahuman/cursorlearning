# Кейс 4: Анализ документации

## Ситуация
К тебе пришёл новый разработчик в команду. Нужно быстро ввести его в курс дела по API. Также ты сам хочешь проверить: покрывает ли текущий API все сценарии для фичи «Сторис», которую ты планируешь.

## Данные
- `data/api_docs_loyalty.md` — документация API LoyalBox v2.3
- `data/feature_brief_stories.md` — бриф на фичу «Сторис»

## Промпт для Cursor

```
Прочитай:
- data/api_docs_loyalty.md — документация API LoyalBox
- data/feature_brief_stories.md — бриф на фичу «Сторис»

Сделай три вещи:

1. **Саммари API для нового разработчика** (файл api_summary.md):
   - Краткое описание каждого эндпоинта (1 строка)
   - Таблица: endpoint, метод, что делает, нужна ли авторизация
   - Основные ограничения (rate limits, лимиты списания и т.д.)
   - Список вебхуков с описанием когда срабатывают

2. **GAP-анализ API для фичи Сторис** (файл stories_api_gap.md):
   - Какие эндпоинты уже есть и подходят для сторис
   - Каких эндпоинтов не хватает (CRUD для сторис, аналитика просмотров, etc.)
   - Предложи структуру новых эндпоинтов (путь, метод, примерный request/response)
   - Приоритизация: что нужно для v1, что для v2
```

## Что демонстрирует
- Cursor анализирует техническую документацию
- Умеет делать кросс-анализ двух документов
- Генерирует саммари разного уровня (для разработчика и для PM)
- Находит пробелы и предлагает решения
- Экономит часы на ручном разборе API
